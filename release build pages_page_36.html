How they work
Almost all the standard reports work either by populating one of the four separate ‘report’ tables within the sched2.mdb database or by setting the field ordheader.report to true for the orders to be reported on and then calling Crystal run time objects to display and provide print and export options. This makes the report simpler – parameters such as work centre and date range can be derived from the planning board and other options selected off the Screenmode menu or by right clicking on a work centre.
When a report is requested PSA takes these steps
  • Populates the report table or flag the orders to be reported – if none are found it does not proceed
  • Add a prefix depending on the page size - A4 or Lt (Letter) to the standard report name
  • Check first in the Custom direct for this report – if not found use the standard report
  • Open the report structure and set the correct database paths for tables used within the report. For PSA tables this will be sched2.mdb in your current working directory. For links to Radius or SFDC it will use the DSN defined under Configuration/System on the ‘parameters’ tab. See ‘Linking to Radius’ below
  • Call Crystal runtime objects to display the report and provide print and export options
